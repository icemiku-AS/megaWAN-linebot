# 小浣 LINE Bot v1.10.2 Secretary Cleanup Edition

這是 **MEGA浣 / 小浣** 的 LINE Bot 專案。

目前專案定位是：以 **Google Apps Script** 為主體的新聞素材秘書與節目準備輔助工具。

小浣不是 Node.js 專案，也不是部署在自架伺服器上的 Bot。主要執行環境為 Google Apps Script，資料儲存以 Google Sheets 為核心。

---

## 1. 專案定位

小浣的核心用途是協助節目製作與日常素材收集，特別是：

- 收集群組或個人聊天室貼上的新聞網址。
- 將網址整理成 NewsInbox 素材池。
- 協助查看最近新聞素材。
- 針對指定網址或近期素材產生節目話題分析。
- 將近期討論整理、統整、封存成長期記憶摘要。

v1.10.2 開始，小浣的產品方向正式收斂為「新聞素材秘書」。

因此，本版移除了幾個低使用率、容易和其他功能重疊的指令，讓入口更乾淨，後續維護也更容易。

---

## 2. v1.10.2 本版重點

### 2.1 指令瘦身

v1.10.2 移除了以下指令：

- `#摘要`
- `#摘要最近`
- `#回顧最近`
- `#標題`
- `#讀網址`

這些指令多半與既有功能重疊，或不符合目前「素材收集秘書」的定位。

例如：

- 最近內容整理可由 `#統整話題` 負責。
- 節目分析可由 `#節目話題分析` 負責。
- 長期摘要封存可由 `#封存本週話題` 負責。
- 快讀網址統一改由 `#懶人包` 負責。
- 標題需求可直接透過一般對話請小浣協助，不再保留獨立指令。

### 2.2 直接貼網址行為統一

v1.10.2 之後，個人聊天室與群組聊天室的網址行為一致。

直接貼網址時：

- 個人聊天室：網址會收進 NewsUrlQueue，背景處理後進 NewsInbox。
- 群組聊天室：網址會收進 NewsUrlQueue，背景處理後進 NewsInbox。

也就是說，**直接貼網址的預設行為是收集素材，不是產生快讀摘要。**

若使用者需要快讀摘要，請明確使用：

```text
#懶人包 網址
```

若使用者需要節目話題分析，請使用：

```text
#節目話題分析 網址
```

這樣可以避免「貼網址到底是要收集、快讀，還是分析」的語意混亂。

### 2.3 個人與群組測試體驗同步

群組裡通常需要 `#小浣` 才會觸發一般對話，但個人聊天室不需要。

不過，網址收集、懶人包、節目分析等核心功能，應該在個人與群組保持相同邏輯。

因此 v1.10.2 的設計原則是：

- 功能行為一致。
- 觸發方式依聊天室情境調整。
- 資料仍依 conversationId 分區，不強制混在一起。

## 3. 目前保留指令

### 3.1 新聞素材收集與查看

#### `直接貼網址`

直接貼上網址時，小浣會將網址加入 NewsUrlQueue，並由背景流程抓取網頁資訊、分類與整理，再寫入 NewsInbox。

這是目前小浣最主要的素材收集入口。

#### `#本週新聞`

查看最近 7 天收集到的新聞素材。

此指令主要用於節目準備前，快速盤點最近有哪些可討論的新聞或素材。

#### `#新聞補充 文字 + 網址`

當某些網頁無法順利抓取，或需要手動補充標題、重點、背景時，可以使用此指令。

用途是把人工整理後的內容補進 NewsInbox，避免好素材因爬取失敗而遺失。

---

### 3.2 網址快讀與節目分析

#### `#懶人包 網址`

針對指定網址產生快讀摘要。

此指令是 v1.10.2 後唯一保留的明確網址快讀入口。

原本的 `#讀網址` 已移除。

#### `#節目話題分析 網址`

針對指定網址產生較深入的節目話題分析。

此功能適合在準備《現正熱潮中》或其他討論型內容時使用。

#### `#節目話題分析`

不附網址時，小浣會根據近期聊天、已整理的網址摘要、WeeklySummary 等資料，判斷適合分析的主題。

---

### 3.3 話題整理與長期記憶

#### `#統整話題`

整合近期對話、網址摘要、WeeklySummary 等資料，整理成節目可用的話題地圖。

這個功能比舊版的 `#摘要最近` 或 `#回顧最近` 更符合目前的小浣定位，因此舊指令已移除。

#### `#封存本週話題`

將近期討論整理成更精簡的 WeeklySummary，作為長期記憶使用。

這個功能適合在一週節目或一輪討論結束後執行。

#### `#記錄 內容`

手動將指定內容寫入 ConversationLog。

適合記錄節目靈感、臨時備忘、重要決策或需要之後回顧的內容。

---

### 3.4 群組對話與管理指令

#### `#小浣 問題`

在群組中呼叫小浣進行一般對話。

個人聊天室不需要 `#小浣`，可以直接輸入問題。

#### `#help`

查看目前可用指令。

#### `#版本`

查看目前版本。

#### `#版本紀錄`

查看近期版本紀錄摘要。

#### `#reset`

清空當前使用者或當前 conversationId 的短期記憶狀態。

#### `#清空紀錄`

要求清空目前 conversationId 的 ConversationLog。

為避免誤刪，這個指令需要二次確認。

#### `#清空紀錄 確認`

確認清空目前 conversationId 的 ConversationLog。

## 4. 已移除指令與替代方式

| 已移除指令 | 移除原因 | 建議替代方式 |
| --- | --- | --- |
| `#摘要` | 與一般對話、統整功能重疊 | 直接請小浣摘要，或使用 `#統整話題` |
| `#摘要最近` | 與 `#統整話題` 重疊 | 使用 `#統整話題` |
| `#回顧最近` | 與 `#統整話題`、`#封存本週話題` 重疊 | 使用 `#統整話題` 或 `#封存本週話題` |
| `#標題` | 使用頻率低，不需要獨立指令 | 直接請小浣幫忙想標題 |
| `#讀網址` | 與 `#懶人包` 完全重疊 | 使用 `#懶人包 網址` |

## 5. 聊天室行為規則

### 5.1 個人聊天室

在個人聊天室中：

- 一般文字會直接進入小浣對話流程。
- 直接貼網址會進入 NewsInbox 收集流程。
- 若要快讀網址，請使用 `#懶人包 網址`。
- 若要深度分析網址，請使用 `#節目話題分析 網址`。

### 5.2 群組聊天室

在群組中：

- 一般文字不會喚醒小浣。
- 需要一般對話時，請使用 `#小浣`。
- 直接貼網址會進入 NewsInbox 收集流程。
- 若要快讀網址，請使用 `#懶人包 網址`。
- 若要深度分析網址，請使用 `#節目話題分析 網址`。

### 5.3 設計理由

群組需要避免小浣過度插話，因此一般對話需要 `#小浣`。

但網址素材收集是小浣的核心工作，因此無論個人或群組，直接貼網址都會進入收集流程。

## 7. 檔案配置

目前主要程式碼仍位於以下 Google Apps Script 分檔：

### `00_Config.gs`

集中放置設定值、版本號、指令前綴、模型參數與資料表名稱。

v1.10.2 已從指令前綴中移除廢止指令，只保留目前支援的入口。

### `01_Main.gs`

LINE webhook 主入口。

負責判斷聊天室類型、訊息內容、指令類型、網址處理與不同功能流程的派發。

v1.10.2 的重要調整之一，是讓個人聊天室直接貼網址時，也與群組一樣進入 NewsInbox 收集流程。

### `02_LineCommands.gs`

LINE 指令解析與輔助函式。

包含：

- 指令解析
- conversationId 產生
- help 文字
- 指令前綴移除
- 記錄模式判斷

v1.10.2 已移除 `#摘要`、`#摘要最近`、`#回顧最近`、`#標題`、`#讀網址` 的解析路徑。

### `04_Storage.gs`

Google Sheets 存取與資料儲存相關功能。

包含對話紀錄、摘要、佇列、封存等資料寫入與讀取。

### `09_DeepSeekService.gs`

DeepSeek API 呼叫層。

負責組裝 request、呼叫模型、處理回應。

v1.10.2 已移除不再使用的 `summary`、`review`、`title` 專用 mode 參數。

### `10_TopicFeatures.gs`

話題分析、統整話題、封存本週話題等進階功能。

這是目前節目準備流程的重要檔案。

### `11_Prompts.gs`

集中管理 prompt。

v1.10.2 已移除不再使用的 `summary`、`review`、`title` prompt。

### `12_ResponseTexts.gs`

集中管理固定回覆文字、版本文字、錯誤訊息與 help 補充文案。

### `13_NewsInbox.gs`

NewsInbox 與 NewsUrlQueue 的核心處理。

負責新聞網址收集、背景整理、分類、狀態更新與本週新聞輸出。

## 8. 開發與維護規則

### 8.1 以 GitHub 最新內容為準

若 GitHub 最新檔案與舊對話、舊記憶、舊上傳檔案矛盾，一律以 GitHub 最新檔案為準。

其中，實際 `.gs` 程式碼優先於 README、CURRENT_VERSION 或 changelog。

### 8.2 修改流程

需要修改程式碼時：

1. 不直接修改 `main`。
2. 從 `main` 建立 feature branch 或 hotfix branch。
3. 在分支上完成修改。
4. 開 Pull Request。
5. 在回覆中整理 PR 編號、修改檔案、修改重點、檢查結果。
6. 不代替維護者 merge PR。

PR 是否合併由維護者處理。

### 8.3 CURRENT_VERSION.md 的讀法

`CURRENT_VERSION.md` 是給 AI 或維護者快速理解目前版本狀態的入口檔。

但若文件中仍保留某個工作分支名稱，而 PR 實際已經合併，應以 `main` branch 最新 commit 作為現行版本來源。

也就是說：

- PR 合併前：可參考 PR branch。
- PR 合併後：以 `main` 最新 commit 為準。

### 8.4 changelog 的讀法

`99_changelog.md` 只作為歷史紀錄。

不可因 changelog 中舊版段落提到某功能，就推定目前仍有該功能。

## 9. v1.10.2 後的產品方向

v1.10.2 的重點不是單純刪功能，而是讓小浣的定位更清楚。

未來新增功能時，建議優先思考：

- 這是否有助於新聞素材收集？
- 這是否有助於節目話題整理？
- 這是否有助於長期記憶與素材回顧？
- 這是否會增加不必要的指令複雜度？

若功能只是偶爾使用，且可以透過一般對話完成，就不一定需要新增成獨立指令。
